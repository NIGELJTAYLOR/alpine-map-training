# Alpine Map Training — companion app

Progressive Web App that delivers Levels 1–3 of the Alpine Map Training
workbook interactively. Built for BASI Alpine Level 4 ISTD candidates and
their trainers.

Live: deployed to Vercel from the `main` branch on every push (auto-deploy).
Currently behind Vercel deployment protection — visible only to the owner.

## What's in v1.5

Visual identity (Carta direction):
- Cream-paper palette (`#F4ECD8`) with slate ink, decorative contour brown
- Fraunces (display) + Inter (UI) + JetBrains Mono (machine codes)
- Custom contour-line motif and PerformOS branding throughout

Content:
- All 66 workbook pages from L1, L2, L3
- 20 schematic diagrams + 9 printable templates
- 50+ glossary terms with cross-links
- 4 OpenTopoMap quiz extracts (C7.1, C7.2, D10.1, D10.2)
- "See also" cross-reference panels at the bottom of every page
- `/about` page documenting the build

Interaction:
- Two interactive quizzes (C7.1, D10.1) with auto-grading + self-mark + error log
- 160 spaced-repetition flashcards with SM-2 scheduling
- Per-page completion + self-check checkboxes that persist
- Confidence ratings + readiness checks captured per-skill

Trainer surface:
- Toggle in `/settings` auto-expands answer keys and reveals trainer notes inline
- Trainer pill in the header reminds you when mode is on

PWA:
- Installable to home screen on iOS / Android / desktop
- Works offline once installed (network-first navigation, cache-first assets)
- PerformOS branded favicon + install icons

## Stack

- **Next.js 16** App Router with TypeScript (Turbopack default)
- **Tailwind CSS v4** with Carta design tokens
- **Velite** for the MDX content pipeline
- **react-markdown** for inline question prompts
- **jimp** for OpenTopoMap composite generation (one-off scripts only)
- Hand-rolled **Service Worker** at `public/sw.js`
- Single-key localStorage progress store (`alpine-map-training:progress`)
- Deployed on **Vercel** — auto-deploys from GitHub `main`

## Day-to-day workflows

### Re-ingesting source content
The workbook source lives outside this repo at
`C:\Users\mrnig_ndtz4tw\OneDrive - performos.ai\Documents\Alpine_Map_Training\`.
The migration script reads from there and writes MDX into `content/`.

```bash
node scripts/migrate-content.mjs all          # re-do everything
node scripts/migrate-content.mjs 1            # just Level 1
node scripts/migrate-content.mjs templates    # just templates
```

The script never writes to the source tree — it's strictly read-only on the
OneDrive directory.

### Editing flashcards
Cards live in human-editable `FLASHCARDS_CONTENT.md` at the project root.
Run `npm run flashcards:build` (or any `npm run build`) to regenerate
`src/data/flashcards.generated.ts`. To produce the .docx review copy,
`npm run flashcards:docx`.

### Editing a quiz
Quiz questions live as JSON: `content/quizzes/L<level>.<page>.json`.
Edit the file and run `npx velite` (or any build) to validate the schema.

### Editing the glossary
Glossary terms are in `src/data/glossary.ts`. Add a new entry to the
`GLOSSARY` array — the `/glossary` page picks it up automatically.

### Re-generating quiz map extracts
```bash
npm run maps:build
```
Fetches OpenTopoMap tiles for the Courchevel area, composites four PNGs
into `public/maps/`. Polite to the tile servers (80 ms between fetches);
within OpenTopoMap's "moderate non-commercial" use terms.

### Local dev
```bash
npm run dev           # http://localhost:3000 — service worker is disabled in dev
npm run build         # production build (runs Velite + flashcards build as prebuild)
npm start             # serve the production build locally
```

### Deploy
`git push origin main` → Vercel auto-deploys in ~90s.

## Architecture notes

- **Static-first.** Every workbook page, level index, quiz, template, glossary
  entry, diagram is statically generated at build time. Interactive bits hydrate
  on the client.
- **Single localStorage key** for everything progress-related. JSON blob,
  versioned (`PROGRESS_VERSION`); on schema mismatch, the loader drops to
  empty.
- **MDX runtime is client-side** because the rendered components include
  client-only widgets (interactive checkboxes). The server still renders
  the initial HTML.
- **Service worker is hand-rolled** because Serwist doesn't run cleanly
  under Next 16 Turbopack. ~150 lines, network-first for navigations,
  cache-first for static assets.
